## 1. บทนำและภาพรวมของระบบ (System Overview)
โครงงานกลุ่ม **WatchMart** เป็นการพัฒนาแพลตฟอร์มพาณิชย์อิเล็กทรอนิกส์ (E-Commerce) สำหรับนาฬิกาหรูระดับพรีเมียม เพื่อตอบโจทย์ธุรกิจในการบริหารจัดการคลังสินค้าที่มีมูลค่าสูงและความปลอดภัยในการทำธุรกรรม โดยออกแบบระบบขึ้นมารอบรับสถาปัตยกรรมแบบแยกส่วน (Decoupled Architecture) และเชื่อมต่อผ่าน RESTful API เพื่อความยืดหยุ่นและการทำงานที่รวดเร็ว

---

## 2. ขั้นตอนการวิเคราะห์ระบบ (System Analysis Phase - SDLC)

ในฐานะนักวิเคราะห์และพัฒนาระบบ ได้กำหนดความต้องการหลักของระบบแบ่งออกเป็น 2 ส่วนหลัก ดังนี้:

### 2.1 ความต้องการเชิงฟังก์ชัน (Functional Requirements)
* **ระบบจัดแบ่งสิทธิ์ (Role-Based Access Control):** กำหนดสิทธิ์การเข้าใช้งานระหว่างลูกค้าทั่วไป (`user`), ผู้จัดการคลังสินค้า (`manager`), และผู้ดูแลระบบสูงสุด (`admin`)
* **ระบบจองนาฬิกาและชำระเงินภายหลัง 24 ชม. (Pay Later):** ลูกค้าสามารถกดสั่งซื้อนาฬิกาเพื่อตัดสต็อกไว้ชั่วคราวก่อน แล้วค่อยกลับมาแนบหลักฐานสลิปการโอนเงินผ่านหน้ารายการสั่งซื้อของฉัน (`My Orders`) ภายใน 24 ชั่วโมงเพื่อป้องกันสินค้าชำระเงินซ้อน
* **ระบบตรวจสอบความถูกต้องสลิป 2 ขั้นตอน (2-Step Verification):** 
  * ขั้นแรก: **ผู้จัดการ** ตรวจเช็คความถูกต้องของภาพสลิปกับธนาคารและกดยอดเงินผ่านขั้นแรก (`manager_approved`)
  * ขั้นที่สอง: **ผู้ดูแลระบบ (Admin)** ตรวจสอบความสอดคล้องของบัญชีร้านและกดยืนยันขั้นสุดท้าย (`confirmed`) เพื่ออนุมัติเตรียมการจัดส่งพัสดุ
* **ระบบแสดงดาวและรีวิวนาฬิกา:** แสดงคะแนนรีวิวเฉลี่ยและจำนวนความคิดเห็นในแต่ละแบรนด์สินค้า โดยสินค้าที่ไม่มีคนรีวิวเริ่มต้นแสดงคะแนนที่ `0.0` ดาว

### 2.2 ความต้องการที่ไม่ใช่เชิงฟังก์ชัน (Non-Functional Requirements)
* **Performance (ประสิทธิภาพ):** เวลาในการตอบสนองเฉลี่ยของ API ต้องรวดเร็ว (ต่ำกว่า 500ms ในการทำงานปกติ)
* **Security (ความปลอดภัย):** ความปลอดภัยในการเข้าถึง API และตรวจสอบสิทธิ์ของผู้ใช้งานในหลังบ้านเพื่อป้องกันผู้ใช้อื่นดักยิง API ข้ามสิทธิ์
* **Reliability & Scalability:** ระบบต้องรองรับปริมาณผู้ใช้ที่ประมวลผลพร้อมกันจำนวนมากได้โดยไม่มีอาการคอขวด

## 3. ขั้นตอนการออกแบบระบบ (System Design Phase)

สถาปัตยกรรมการวิเคราะห์และออกแบบระบบ ถูกวาดรายละเอียดผ่าน UML Diagrams ดังนี้:

### 3.1 แผนภาพ Use Case (Use Case Diagram)
ใช้แสดงการโต้ตอบของผู้ใช้งานกับระบบเพื่อระบุขอบเขตการทำงาน (System Boundary):
* **ผู้ซื้อ (Customer):** ค้นหาสินค้า, จัดการตะกร้า, สั่งซื้อสินค้า (Pay Later), แนบหลักฐานสลิป และติดตามสถานะออเดอร์
* **ผู้จัดการ (Manager):** จัดการข้อมูลนาฬิกาในสต็อก, อนุมัติสลิปขั้นแรก, และบันทึกจัดส่งสินค้า
* **แอดมิน (Admin):** อนุมัตินำสินค้าเข้าขาย, ยืนยันยอดเงินขั้นสุดท้าย (Final Confirm), และตรวจสอบบันทึกระบบ (System Logs)

### 3.2 แผนภาพคลาส (Class Diagram)
แสดงโครงสร้างข้อมูลเชิงวัตถุและความสัมพันธ์ระหว่างตาราง:
* คลาส **Product** เชื่อมโยงกับคลาส **Review** เพื่อคำนวณสถิติดาวเฉลี่ย
* คลาส **User** เชื่อมโยงการครอบครองกับคลาส **Cart** และสร้างความสัมพันธ์แบบหนึ่งต่อกลุ่ม (1 to Many) ไปยังคลาส **Order**
* คลาส **Order** เก็บข้อมูลหลักฐานโอนเงินในฟิลด์ **slip** และสอดคล้องกับคลาสสถานะ flow การอนุมัติ 2 ขั้นตอน

### 3.3 แผนภาพขั้นตอนการทำงาน (Sequence Diagram)
อธิบายขั้นตอนส่งข้อความการตรวจสอบสลิป 2 ขั้นตอน:
1. ลูกค้า Checkout -> เซิร์ฟเวอร์สร้างใบจอง (`pending_payment`)
2. ลูกค้าแนบภาพสลิป -> สถานะเปลี่ยนเป็น `pending_review`
3. ผู้จัดการตรวจสอบ -> ส่งคำขอยืนยัน -> สถานะเปลี่ยนเป็น `manager_approved`
4. แอดมินกดยืนยันสุดท้าย -> ส่งคำขออนุมัติสำเร็จ -> สถานะเปลี่ยนเป็น `confirmed`

## 4. ขั้นตอนการพัฒนาระบบและการทดสอบ (System Development & Testing)

### 4.1 เทคโนโลยีที่เลือกใช้ (Tech Stack)
* **Frontend:** React + Vite + Vanilla CSS ในการสร้างอินเตอร์เฟซที่ตอบสนองอย่างรวดเร็ว (Responsive UI)
* **Backend:** Node.js (TypeScript) + Express ในการเขียน API เชื่อมต่อแบบ RESTful
* **Database:** PostgreSQL (เซิร์ฟเวอร์แบบ Serverless ของ Neon) ทำงานร่วมกับ PostgreSQL Connection Pooling เพื่อประสิทธิภาพในการอ่านเขียนข้อมูลที่รวดเร็ว

### 4.2 แผนการทดสอบคุณภาพ (Testing Plan)
* **การทดสอบฟังก์ชันการทำงาน (API Testingด้วย Postman):** 
  * จัดทำ Collection รวบรวมคำร้องขอทุก Endpoint (เช่นการล็อกอินสมัครสมาชิก และโฟลว์ตรวจสอบสลิป 2 ขั้น) พร้อมตั้งคำสั่ง Test Script เพื่อยืนยันความถูกต้องของ JSON Schema และค่าผลลัพธ์ตอบกลับจากเซิร์ฟเวอร์
* **การทดสอบขีดความสามารถการรับโหลด (Load Testing ด้วย JMeter):**
  * จำลองเหตุการณ์ผู้ใช้งานพร้อมกัน 10, 50 และ 100 Threads ค่อยๆ ทยอยปล่อยเข้าสู่ระบบ (Ramp-up 5 วินาที)
  * ผลลัพธ์แสดงให้เห็นว่าอัตราการสูญเสียคำขอมีค่าเท่ากับ **0.00% (Error Rate)** และ Throughput ขยายตัวขึ้นแบบแปรผันตามจำนวนผู้ใช้อย่างมีประสิทธิภาพ

## 5. สรุปบทบาทการเรียนรู้และการนำทฤษฎีไปใช้งาน
จากการพัฒนากรณีศึกษา **WatchMart** ทำให้เข้าใจหลักการทำงานของ **SDLC** ตลอดวงจร ตั้งแต่การวิเคราะห์ฟังก์ชันจนถึงการประเมินประสิทธิภาพหลังบ้านด้วยโหลดเทส ซึ่งช่วยลดระยะเวลาความซ้ำซ้อนของการประมวลผลด้วยกลไก Caching และยืนยันความปลอดภัยในระบบธุรกรรมการเงินด้วยระบบยืนยันสลิป 2 ขั้นตอนอย่างแท้จริง
