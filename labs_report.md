## 📡 Lab #1: การทดสอบ API ด้วย Postman

### 1. ทำการสร้าง Collection และ HTTP เพื่อทดลองดึงค่าต่างๆ จากปลายทาง
ทำการทดลองส่งคำขอ (Request) ไปยัง Public API ปลายทาง: `https://jsonplaceholder.typicode.com/posts` เพื่อวิเคราะห์ผลลัพธ์ผ่านคำสั่ง GET Request

* **วิเคราะห์ผลลัพธ์:** หลังจากกดส่งคำสั่ง GET Request ข้อมูลทั้งหมดจาก `/posts` จะถูกดึงมาแสดงในส่วนของ Body ในรูปแบบข้อมูลชนิด JSON โดยประกอบไปด้วยคีย์หลัก ได้แก่ `"userId"`, `"id"`, `"title"`, `"body"`
* **ผลลัพธ์จากระบบ:**
  * **Status:** `200 OK` (หมายถึงเซิร์ฟเวอร์ตอบกลับการทำงานเป็นปกติ)
  * **Time:** `136 ms` (ความเร็วในการตอบสนองและแสดงผลข้อมูล)
  * **Size:** `8.02 KB` (ขนาดของข้อมูล JSON)
* **วิธีการดึงข้อมูลหลัก 2 วิธี:**
  1. **การดึงข้อมูลแบบทั้งหมด (Get All):** เรียกใช้งานผ่าน `GET https://jsonplaceholder.typicode.com/posts`
  2. **การดึงข้อมูลแบบระบุรายการ (Get by ID):** เรียกใช้งานผ่าน `GET https://jsonplaceholder.typicode.com/posts/1`

---

### 2. ทดลองส่ง POST Request พร้อม Body (JSON)
ส่งข้อมูลใหม่เข้าไปบันทึกในระบบผ่านคำสั่ง POST Request ไปยังปลายทาง `/posts` โดยกำหนดข้อมูลในรูปแบบ Array of Objects ดังนี้:

* **Request Body (JSON):**
  ```json
  [
    {
      "userId": 1,
      "title": "Hello API [..Postman..] 1",
      "body": "This is a test POST request 1!"
    },
    {
      "userId": 2,
      "title": "Hello API [..Postman..] 2",
      "body": "This is a test POST request 2!"
    }
  ]
  ```
* **วิเคราะห์ผลลัพธ์:** การส่งคำสั่ง POST เพื่อเพิ่มข้อมูลแบบ Array ของ Object ระบบจะทำการสร้างข้อมูลชุดใหม่ขึ้นมาและส่งคืนค่าคุณสมบัติ `"id": 101` (ไอดีชุดข้อมูลใหม่ลำดับถัดจากข้อมูลเดิมที่มีอยู่ 100 ชุด)
* **ข้อมูลประสิทธิภาพที่ได้:**
  * **Status:** `201 Created` (ระบุว่าระบบได้สร้างข้อมูลใหม่สำเร็จแล้ว)
  * **Time:** `621 ms`
  * **Size:** `1.45 KB`

---

### 3. ทดลองสร้าง Test Script เพื่อตรวจสอบค่าตามเงื่อนไข

#### 3.1 ตรวจสอบว่า Response Status Code ต้องเป็น 201 (Created)
```javascript
pm.test("Status Code should be 201", function () {
    pm.response.to.have.status(201);
});
```
* **ผลการทดสอบ:** **ผ่าน (PASS)** เนื่องจากเซิร์ฟเวอร์ส่งรหัสการทำงานกลับมาเป็น `201 Created` ตรงตามเงื่อนไข

#### 3.2 ตรวจสอบว่า Response เป็น JSON format
```javascript
pm.test("Response must be in JSON format", function () {
    pm.response.to.be.json;
});
```
* **ผลการทดสอบ:** **ผ่าน (PASS)** เนื่องจากระบบตรวจสอบ Header ของ Response และพบว่าชนิดข้อมูลส่งกลับมาเป็นแอปพลิเคชัน JSON จริง (หากเป็น HTML จะล้มเหลว)

#### 3.3 ตรวจสอบ Response Body ค่า Title ของออบเจกต์แรก
```javascript
pm.test("Title should match the input", function () {
    var jsonData = pm.response.json();
    pm.expect(jsonData[0].title).to.eql("Hello API [..Postman..] 1");
});
```
* **วิเคราะห์ข้อผิดพลาดและวิธีแก้ไข:** หากเขียน `pm.expect(jsonData.title)` ผลการทดสอบจะ **ไม่ผ่าน** เนื่องจากข้อมูลที่ส่งกลับมาเป็น Array ไม่ใช่ Object เดี่ยว ดังนั้น วิธีการเข้าถึงข้อมูลที่ถูกต้องจึงต้องกำหนดดัชนีเป็น `jsonData[0].title` เพื่อเข้าถึง Object ตัวแรกใน Array ผลการทดสอบจึงจะ **ผ่าน (PASS)**

#### 3.4 ตรวจสอบว่า User ID ของออบเจกต์ที่สองต้องเท่ากับ 2
```javascript
pm.test("Check second object's userId", function () {
    var jsonData = pm.response.json();
    pm.expect(jsonData[1].userId).to.equal(2);
});
```
* **ผลการทดสอบ:** **ผ่าน (PASS)** เนื่องจากดัชนีอาร์เรย์ตำแหน่งที่ 1 (`jsonData[1]`) มีค่าคุณสมบัติ `userId` เท่ากับ 2 ตรงตามค่าที่ป้อนส่งไป

#### 3.5 ตรวจสอบพฤติกรรมการตอบกลับในเรื่องของเวลา (Response Time) ต้องน้อยกว่า 2 วินาที (2000ms)
```javascript
pm.test("Response time should be less than 2000ms", function () {
    pm.expect(pm.response.responseTime).to.be.below(2000);
});
```
* **ผลการทดสอบ:** **ผ่าน (PASS)** ความเร็วในการตอบสนองจริงอยู่ที่ `315 ms` (หรือเท่ากับ `0.315 วินาที`) ซึ่งน้อยกว่าขีดจำกัดความเร็ว 2 วินาทีที่ระบบตั้งไว้

### 4. ทดสอบ API ด้วย PUT / DELETE

* **Method PUT (การแก้ไขข้อมูล):**
  * ใช้สำหรับการแก้ไขหรืออัปเดตข้อมูลเดิมที่มีอยู่ในระบบ เช่น ส่งคำร้องขอไปยัง `PUT /posts/1` เพื่อแก้ไขฟิลด์ Title ของข้อมูลเดิม
  * **วิเคราะห์ผลลัพธ์:** ข้อมูลภายใน Object ที่ระบุจะถูกอัปเดตและแทนที่ด้วยค่าใหม่ที่เราป้อนส่งไป
* **Method DELETE (การลบข้อมูล):**
  * ใช้สำหรับการลบข้อมูลออกจากฐานข้อมูลระบบ เช่น ส่งคำร้องขอไปยัง `DELETE /posts/1`
  * **วิเคราะห์ผลลัพธ์:** ระบบดำเนินการลบข้อมูลในตำแหน่ง ID 1 และส่งรหัสการทำงานสำเร็จกลับคืนมา

#### 4.1 การใช้งาน Environment Variables ใน Postman
* **ความหมาย:** การสร้างกล่องจัดเก็บตัวแปรระบบ เช่น กำหนดตัวแปร `{{baseUrl}}` แทนชื่อเซิร์ฟเวอร์ปลายทาง เพื่อช่วยลดความซ้ำซ้อนในการพิมพ์ URL เดิมซ้ำ ๆ และเอื้อให้สามารถกดสลับสภาพแวดล้อมการทดสอบ (เช่น จาก Dev -> Staging -> Production) ได้อย่างรวดเร็วโดยไม่ต้องแก้ไข URL ของทุกคำขอแบบแมนนวล

## ⚡ Lab #2: การทดสอบประสิทธิภาพ (Load Testing) ด้วย JMeter

### 1. โครงสร้างการทดสอบ
ทำการสร้างแผนการทดสอบ (Test Plan) ใน Apache JMeter โดยประกอบไปด้วยองค์ประกอบหลักดังนี้:
* **Thread Group** (กำหนดจำนวนและพฤติกรรมผู้ใช้จำลอง)
* **HTTP Request** (กำหนดปลายทาง API ที่จะทำการยิงทดสอบ)
* **Summary Report** (รายงานสรุปสถิติตัวเลขประสิทธิภาพ)
* **View Results Tree** (รายละเอียดคำตอบรับของแต่ละคำขอ)
* **Graph Results** (กราฟแสดงแนวโน้มประสิทธิภาพภายใต้ภาระโหลด)

---

### 2. การตั้งค่า Thread Group
* **Number of Threads (Users):** กำหนดจำนวนผู้ใช้งานที่จะจำลองส่งคำร้องขอพร้อมกัน (เช่น 10 คน)
* **Ramp-Up Period (seconds):** ระยะเวลาเฉลี่ยในการปล่อยจำนวนผู้ใช้ เช่น หากตั้งไว้ 5 วินาทีกับผู้ใช้ 10 คน JMeter จะทยอยส่งผู้ใช้เข้ามาวินาทีละ 2 คนจนครบในวินาทีที่ 5
* **Loop Count:** จำนวนรอบในการทำงานของผู้ใช้แต่ละคน เช่น ตั้ง Loop เป็น 1 หมายถึงส่งคนละ 1 ครั้ง หากตั้งเป็น 10 หมายถึงแต่ละคนจะส่งคำขอซ้ำ 10 รอบ

---

### 3. ผลการทดสอบและสถิติประสิทธิภาพ (Performance Metrics Summary)

ตารางสรุปข้อมูลสถิติที่ได้จากการทดสอบระบบภายใต้การเพิ่มจำนวนผู้ใช้งานจำลอง (Threads) ระดับ 10, 50, และ 100 คน:

| จำนวนผู้ใช้งานจำลอง (Threads) | จำนวนคำขอทั้งหมด (# Samples) | เวลาตอบกลับเฉลี่ย (Average - ms) | เวลาตอบกลับช้าที่สุด (Max - ms) | อัตราข้อผิดพลาด (Error %) | อัตราการประมวลผลสำเร็จ (Throughput) |
|:---:|:---:|:---:|:---:|:---:|:---:|
| **10 คน** | 10 | 608 ms | 1,030 ms | 0.00% | 2.0 / sec |
| **50 คน** | 50 | 460 ms | 981 ms | 0.00% | 9.5 / sec |
| **100 คน** | 100 | 410 ms | 480 ms | 0.00% | 18.6 / sec |

---

### 4. การวิเคราะห์ประสิทธิภาพระบบ (Performance Analysis)

#### 4.1 ความเสถียรของระบบ (System Stability)
* **จุดสังเกต:** อัตราข้อผิดพลาด (Error %)
* **ผลลัพธ์และการวิเคราะห์:** ในทุกรอบการทดสอบตั้งแต่โหลดน้อยไปจนถึงโหลดสูงสุด ค่า Error % มีค่าเท่ากับ **0.00%** เสมอ หมายความว่าระบบมีความเสถียรภาพและคุณภาพสูงมาก สามารถรองรับปริมาณคำขอและทำงานประมวลผลได้อย่างถูกต้องครบถ้วน 100% โดยไม่มีเหตุการณ์ข้อมูลสูญหาย คำร้องขอตกหล่น หรือเซิร์ฟเวอร์ล่ม (เช่น เกิดรหัส 500 Internal Server Error) แม้จะมีผู้ใช้งานเพิ่มขึ้นเป็น 100 คนพร้อมกัน

#### 4.2 ความสามารถในการขยายตัวและรองรับการเพิ่มโหลด (Scalability)
* **จุดสังเกต:** อัตราการประมวลผล Throughput (จำนวนคำขอที่จัดการได้สำเร็จต่อวินาที)
* **ผลลัพธ์และการวิเคราะห์:** อัตรา Throughput มีแนวโน้มเติบโตขึ้นเป็นเส้นตรงแบบแปรผันตามสัดส่วนผู้ใช้ (Linear Scalability) โดยขยับจาก **2.0/sec** ขึ้นเป็น **9.5/sec** และทะยานถึง **18.6/sec** ตามลำดับ แสดงว่าเซิร์ฟเวอร์และโครงสร้างฐานข้อมูลยังมีทรัพยากรรองรับที่เพียงพอ สามารถรีดประสิทธิภาพการทำงานเพื่อตอบสนองปริมาณโหลดจำนวนมากได้ดีโดยไม่มีอาการคอขวด (Bottleneck) ให้เห็น

#### 4.3 พฤติกรรมการตอบกลับของระบบ (Response Behavior under Load)
* **จุดสังเกต:** เวลาตอบรับเฉลี่ย (Average) และเวลาตอบรับช้าที่สุด (Max)
* **ผลลัพธ์และการวิเคราะห์:** เกิดพฤติกรรมที่น่าสนใจคือยิ่งปริมาณโหลดหนาแน่นขึ้น เวลาตอบสนองเฉลี่ยกลับ **ลดลงและประมวลผลเร็วขึ้น** (จาก 608 ms ลดเหลือ 410 ms) รวมถึงเวลาตอบรับที่ช้าที่สุด (Max) ก็บีบแคบลงอย่างเห็นได้ชัดจาก 1,030 ms เหลือเพียง 480 ms พฤติกรรมนี้ชี้ให้เห็นว่าระบบทำงานร่วมกับกลไก **Connection Pooling** หรือมีระบบ **Caching** ข้อมูลที่มีประสิทธิภาพสูง ทำให้เมื่อมีคำร้องขอซ้ำ ๆ ไหลเข้ามาอย่างต่อเนื่อง ระบบจึงสามารถเรียกค้นข้อมูลจากหน่วยความจำแคชเพื่อส่งกลับไปตอบสนองผู้ใช้ได้ทันทีโดยไม่ต้องไปทำการเชื่อมต่อฐานข้อมูลใหม่ทุกครั้ง

### 5. สรุปผลการประเมิน
ระบบผ่านการทดสอบประสิทธิภาพการรับโหลด (Load Testing) ในระดับ 100 Threads (ผู้ใช้งานพร้อมกัน 100 คน) ได้อย่างสมบูรณ์แบบ โครงสร้างระบบยังคงเสถียรภาพ ทำงานได้ถูกต้อง และประมวลผลรวดเร็วตามเกณฑ์วิศวกรรมซอฟต์แวร์
